# /improve-architecture — 架构深化

寻找架构摩擦点，提出**深化机会** — 将浅模块变为深模块的重构建议。目标是提升可测试性和 AI 可导航性。

## 核心概念

| 术语 | 定义 |
|---|---|
| **模块** | 任何有接口和实现的单元（函数、类、包、服务） |
| **接口** | 调用者必须了解的一切：类型、不变量、错误模式、配置 |
| **深模块** | 小接口背后隐藏大量复杂性 — 高杠杆 |
| **浅模块** | 接口几乎和实现一样复杂 — 低杠杆 |
| **删除测试** | 想象删除该模块：如果复杂性消失，它是传递层；如果复杂性散布到 N 个调用者，它在创造价值 |

## 执行步骤

### Step 1：探索与识别摩擦

读取项目的 `ARCHITECTURE.md` 和 `AGENTS.md` 了解整体架构，然后探索代码库，注意以下信号：

**浅模块信号**：
- 理解一个概念需要在多个小文件间跳转
- 函数只是简单转发到另一个函数（pass-through）
- 接口定义几乎和实现一样长
- 为了可测试性提取的纯函数，但真正的 bug 藏在调用方式中

**深化机会信号**：
- 多个调用者重复相同的组合逻辑
- 复杂性散布在多个模块而非集中在一处
- 紧耦合的模块跨越了本应清晰的边界
- 难以测试的代码（没有好的测试接缝）

对可疑模块应用**删除测试**：删除它后，复杂性会集中还是散布？

### Step 2：呈现候选方案

输出编号列表，每个候选包含：

```markdown
## 深化机会 #N：[名称]

**涉及文件**：
- `path/to/file1.go`
- `path/to/file2.go`

**当前问题**：[为什么当前架构造成摩擦]

**建议方案**：[一段话描述如何深化]

**收益**：
- 局部性：[变更/bug/知识如何集中到一处]
- 杠杆：[调用者获得什么简化]
- 可测试性：[测试如何改善]
```

**不提议具体接口设计** — 只描述方向。询问用户：「你想深入探讨哪个方案？」

### Step 3：与用户交互确认

用户选择候选后，进入讨论：

- 约束条件是什么？
- 深化后的模块边界在哪？
- 什么是接缝（seam）？什么是适配器？
- 哪些测试会因为这次重构而变好？
- 是否与现有 `docs/decision-log.md` 中的 ADR 冲突？

如果用户拒绝某方案且理由有建设性，建议记录为 ADR 防止未来重复建议。

### Step 4：输出重构计划

确认方案后，输出可执行的重构步骤：

```markdown
## 重构计划：[名称]

### 前置条件
- [ ] 现有测试全部通过
- [ ] 相关模块的覆盖率 >= X%

### 步骤
1. [步骤描述] — 预期影响：[文件列表]
2. ...

### 验证方式
- [ ] 重构后所有测试通过（不修改断言）
- [ ] 新增的测试覆盖了深化后的接口
- [ ] 调用者代码变简单了（diff 负增长）

### 风险
- [风险描述及缓解方式]
```

## 禁止项

- 不在没有具体摩擦点的情况下建议重构（"代码看起来可以更好"不是有效理由）
- 不建议用户拆分没有实际问题的模块（如果没坏，不修）
- 不在修复 bug 时顺便重构（保持 diff 干净，重构单独做）
- 不提议与现有 ADR 冲突的方案（除非摩擦大到值得重新讨论 ADR）
- 不自行执行重构 — 必须先获得用户确认
