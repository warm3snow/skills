# /spec-write — SpecKit 设计文档生成

你是团队技术方案的起草专家。引导用户完成大型变更的完整 SpecKit 流程，产出可评审的规范文档。

## 触发条件判断

在开始前，先判断变更规模：

| 规模 | 判断标准 | 适用流程 |
|------|----------|----------|
| **大型** | 新模块、新数据库表、跨 2+ 模块改动 | SpecKit 全流程（spec + clarify + plan + tasks） |
| **中型** | 现有模块内新增/修改功能 | spec.md + plan.md，无需 clarify 和 tasks |
| **小型** | Bug 修复、配置调整、单模块内改动 | 直接用 `/new-feature`，不需要 spec |

**升级条款**：执行中发现范围超出当前级别，必须暂停升级，不可硬扛。

## 执行前准备

读取以下文件（如存在）：
1. `AGENTS.md` — 项目约定、技术栈
2. `ARCHITECTURE.md` — 现有架构，避免方案冲突
3. `harness/api-standards.md` — API 设计规范
4. `harness/database.md` — 数据库规范（如涉及）
5. `harness/testing.md` — 测试策略要求

## 执行步骤

### Step 1：需求澄清（clarify）

在起草文档前，必须明确以下问题：

**功能边界**
- 核心功能是什么？用一句话描述"这个功能让用户能做 xxx"
- 明确不做什么（Out of Scope）
- 与哪些现有模块交互？

**业务规则**
- 有哪些边界条件和异常场景？
- 数据验证规则是什么？
- 权限控制要求？

**验收标准**
- 功能完成的判断标准是什么？
- 性能要求（响应时间、并发量）？
- 是否需要向后兼容？

将澄清结果保存到 `specs/<编号>/clarify.md`。

### Step 2：生成 spec.md

文件路径：`specs/<编号>/spec.md`

```markdown
# Spec: <功能名称>

**编号**：SPEC-XXXX
**状态**：Draft / In Review / Approved
**作者**：（研发）
**创建时间**：YYYY-MM-DD
**关联单号**：[TAPD-XXXX]

---

## 1. 背景与目标

<!-- 为什么要做这个？解决什么问题？ -->

## 2. 功能描述

<!-- 核心功能，用用户视角描述 -->

## 3. Out of Scope

<!-- 明确不做什么，避免范围蔓延 -->

## 4. 边界条件 & 异常场景

<!-- 列举所有需要特殊处理的情况 -->

## 5. 验收标准

<!-- 可量化、可验证的验收条件 -->
- [ ] 验收项 1
- [ ] 验收项 2

## 6. 依赖与风险

<!-- 上下游依赖、技术风险、不确定性 -->
```

### Step 3：生成 plan.md（架构方案）

文件路径：`specs/<编号>/plan.md`

```markdown
# Plan: <功能名称>

## 架构方案

<!-- 模块划分、数据流、接口设计 -->

## 数据模型变更

<!-- 新增/修改的表结构、字段说明 -->

## API 设计

<!-- 新增/修改的接口，符合 harness/api-standards.md -->

## 技术选型

<!-- 关键技术决策及理由，特别说明"为什么选这个而不是另一个" -->

## 已知技术债

<!-- 本方案引入的技术债，后续需处理 -->
```

### Step 4：生成 tasks.md（任务拆解）

文件路径：`specs/<编号>/tasks.md`

规则：
- 每个子任务必须可在 1 天内完成
- 每个子任务一个 PR
- 包含 TDD 步骤（先写测试再实现）
- 标注前置依赖

```markdown
# Tasks: <功能名称>

## 子任务列表

- [ ] TASK-01: [描述] — 前置：无 — 产出：xxx
- [ ] TASK-02: [描述] — 前置：TASK-01 — 产出：xxx

## TDD 执行顺序

每个 TASK 的执行顺序：
1. 写接口/函数签名（不实现）
2. 写测试（确保失败）
3. 实现代码（确保通过）
4. 重构（保持测试通过）
```

## 禁止项

- 不在需求未澄清的情况下直接生成 plan.md
- 不创建无法在 1 天内完成的子任务
- 不在 spec 中省略 Out of Scope 和验收标准
- 不省略技术选型的理由（"为什么"比"是什么"更重要）
