# /prototype — 一次性原型验证

原型是**回答问题的临时代码**。问题决定了原型的形态。

## 核心原则

1. **从第一天就是临时的** — 命名清晰标注为原型，不是正式代码
2. **一条命令运行** — 用项目已有的任务运行器
3. **默认不持久化** — 状态在内存中，除非问题本身涉及持久化
4. **跳过打磨** — 不写测试、不做错误处理（只保证能运行）、不做抽象
5. **展示状态** — 每次操作后打印/渲染完整相关状态
6. **用完即删或吸收** — 回答了问题后，删除或将验证通过的决策融入正式代码

## 判断分支

先识别要回答的问题：

| 问题类型 | 分支 | 原型形态 |
|---|---|---|
| 这个逻辑/状态模型对不对？ | **逻辑原型** | 交互式终端应用，推动状态机走过难以纸上推演的场景 |
| 这个接口/数据流设计行不行？ | **逻辑原型** | 最小化子系统，用单个函数调用验证设计 |
| 这个 UI/交互方案好不好？ | **UI 原型** | 多个差异化方案在同一路由切换对比 |

## 执行步骤

### Step 1：明确要回答的问题

用一句话写清：「这个原型要回答的问题是：xxx？」

如果问题模糊，先与用户澄清。一个原型只回答一个问题。

### Step 2：选择分支并构建

#### 逻辑原型

```
原型位置：靠近要验证的模块（如 internal/service/order_prototype/）
运行方式：go run ./cmd/prototype/ 或 make prototype
输出方式：每步操作后打印完整状态变化
```

**构建要点**：
- 聚焦状态转换和边界条件
- 用硬编码数据代替外部依赖
- 提供交互式输入（命令行菜单或预设场景）
- 每次操作后打印 before → after 状态对比

#### UI 原型

```
原型位置：pages/prototype/ 或 components/prototype/
运行方式：与项目正常启动方式一致
切换方式：URL 参数或浮动切换栏
```

**构建要点**：
- 生成 3-5 个差异化明显的方案
- 方案之间的差异要有实质意义，不是配色/字号这种表面差异
- 提供一键切换，方便对比
- 用 mock 数据，不接真实后端

### Step 3：运行验证

运行原型，向用户展示结果。引导用户回答：

- 这个原型是否回答了最初的问题？
- 答案是什么？（用一句话总结结论）
- 是否需要修改方案后再次验证？

### Step 4：记录结论并清理

**记录答案**（在 commit message 或 `docs/decision-log.md` 中）：

```markdown
## [YYYY-MM-DD] 原型验证：<问题简述>

- **问题**：xxx？
- **结论**：xxx
- **验证方式**：xxx
- **对正式实现的影响**：xxx
```

**清理原型代码**：
- 删除原型目录/文件
- 或将验证通过的核心逻辑融入正式代码（去掉原型标记）

## 禁止项

- 不在没有明确问题的情况下写原型（"先写写看"不是有效触发）
- 不把原型代码留在仓库中腐烂（用完必须删除或吸收）
- 不在原型中追求代码质量（临时代码不需要测试和抽象）
- 不用原型替代正式的 spec 设计流程（原型回答设计问题，不替代设计）
- 不让原型依赖生产环境数据或服务
