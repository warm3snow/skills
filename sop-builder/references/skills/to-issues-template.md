# /to-issues — 计划转任务单（垂直切片拆解）

将设计文档（spec）或实施计划拆解为独立可抓取的任务单。每个任务是一个**垂直切片（tracer bullet）**，端到端贯穿所有集成层。

## 核心原则

- **垂直切片**：每个任务贯穿所有层（schema → service → handler → 测试），而不是按水平层拆分
- **独立可验证**：完成一个任务后，该功能路径可以独立演示或验证
- **薄切片优先**：宁可多个薄切片，不要少数厚切片
- **明确依赖关系**：标注阻塞关系，先做被依赖的任务

## 执行步骤

### Step 1：获取上下文

从以下来源获取计划：
- 用户提供的 spec 文档（`specs/` 目录）
- 当前对话中的设计方案
- 已有的 PR 描述或 issue 内容

读取 `AGENTS.md` 和 `ARCHITECTURE.md` 了解项目约定和模块划分。

### Step 2：拆解为垂直切片

将计划拆为 tracer bullet 任务。每个任务必须满足：

- 切穿**所有相关层**（不是只改一层）
- 完成后可**独立验证**
- 包含对应的**测试**
- 工作量控制在 **0.5-1 天**以内

分类：
- **AFK**（可自动执行）：需求明确、不需人工决策、AI 或开发者可直接实现
- **HITL**（需人工介入）：涉及架构决策、设计评审、外部确认

优先 AFK，减少 HITL。

### Step 3：呈现拆解方案

```markdown
## 任务拆解方案（共 N 个任务）

| # | 标题 | 类型 | 阻塞关系 | 涉及层 |
|---|---|---|---|---|
| 1 | [任务标题] | AFK | 无 | schema + service + handler + test |
| 2 | [任务标题] | AFK | 依赖 #1 | service + handler + test |
| 3 | [任务标题] | HITL | 无 | 架构决策 |
```

### Step 4：与用户确认

向用户确认：
- 粒度是否合适？（太粗 / 太细）
- 依赖关系是否正确？
- HITL 和 AFK 分类是否正确？
- 是否需要合并或拆分某些任务？

迭代直到用户批准。

### Step 5：输出任务单

按依赖顺序（被阻塞者优先）输出每个任务的完整描述：

```markdown
---

## 任务 #N：[标题]

**类型**：AFK / HITL
**阻塞关系**：依赖 #X / 无
**关联单号**：[TAPD-XXXX]（如有）

### 目标

一段话描述这个垂直切片端到端要实现什么。描述行为，不描述实现细节。

### 验收标准

- [ ] 标准 1（可验证的具体行为）
- [ ] 标准 2
- [ ] 标准 3
- [ ] 对应测试通过

### 涉及模块

- `path/to/schema_or_migration`
- `path/to/service`
- `path/to/handler`
- `path/to/test`

### TDD 执行顺序

1. 写接口签名（不实现）
2. 写测试（确认失败）
3. 实现代码（确认通过）
4. 提交

---
```

## 禁止项

- 不按水平层拆分（"任务 1: 建表，任务 2: 写 service，任务 3: 写 handler"是错误的）
- 不创建无法在 1 天内完成的任务
- 不省略验收标准（每个任务必须有可验证的完成条件）
- 不在用户未确认拆解方案的情况下直接输出任务单
- 不忽略依赖关系（并行可执行的任务要标明，有先后顺序的要标明）
