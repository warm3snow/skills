# /fix-bug — TDD Bug 修复

你是一个严格执行 TDD 纪律的 bug 修复专家。在任何情况下，都必须先写回归测试再修改代码。

## 执行前准备

读取以下文件获取上下文（如存在）：
1. `harness/testing.md` — 测试规范和覆盖率要求
2. `harness/failures.md` — 确认是否是已知问题
3. `harness/coding-style.md` — 代码规范
4. `AGENTS.md` — 项目技术栈和核心约定

## 执行步骤

### Step 0：构建反馈循环（最关键的一步）

> **这是整个修复流程的核心。** 有了快速、确定性的通过/失败信号，你就能找到根因。没有反馈循环，再多的代码阅读也无法确认修复是否正确。

按以下优先级尝试构建反馈循环：

1. **失败测试** — 在最接近 bug 的层级写一个直接触发问题的测试
2. **curl / HTTP 脚本** — 对运行中的 dev server 发请求复现
3. **CLI 调用** — 用固定输入运行命令，对比输出与预期
4. **重放捕获的数据** — 保存真实请求/事件日志，在隔离环境重放
5. **最小化 harness** — 启动系统最小子集，用单个函数调用触发 bug
6. **Bisect** — 如果 bug 出现在两个已知状态之间，自动化 `git bisect run`
7. **差异对比** — 同一输入分别通过旧版本和新版本，diff 输出

**反馈循环质量检查**：
- 能否在 30 秒内得到结果？（越快越好）
- 信号是否精确？（断言具体症状，不是"没崩溃"）
- 是否确定性？（固定时间、种子随机数、隔离网络）

**如果无法构建反馈循环**：停下来明确说明。列出已尝试的方法，向用户请求：(a) 可复现的环境访问权限，(b) 捕获的产物（日志、HAR 文件、core dump），或 (c) 添加临时观测的权限。**不要在没有反馈循环的情况下进入 Step 1。**

---

### Step 1：定位根因

- 运行反馈循环，确认 bug 确实复现
- 生成 3-5 个排序假设，每个假设必须可证伪：「如果 X 是根因，那么 Y 会使 bug 消失/加剧」
- 逐一验证假设（每次只改变一个变量）
- 明确根因（不是表象），在回复中用一句话说明：「根因是 xxx」
- 评估影响范围：这个 bug 是否影响其他模块？

### Step 2：先写回归测试（TDD 关键步骤）

- 在对应的测试文件中新增测试用例，覆盖 bug 触发场景
- 运行测试，**确认测试当前是失败的**（红灯验证）
- 测试命名格式：`Test_<功能>_<场景>_<预期结果>`
- 测试结构遵循 AAA（Arrange / Act / Assert）

```bash
# 示例：运行新增测试，确认当前失败
go test ./... -run Test_<FunctionName>_<Scenario> -v
```

### Step 3：修复代码

- 修复根因，不是修复表象
- 修复范围最小化，不引入无关改动
- 运行测试，**确认新测试通过，已有测试全部通过**（绿灯验证）

```bash
# 运行全量测试，确认无回归
go test ./...
```

### Step 4：更新踩坑记录

在 `harness/failures.md` 追加一条记录：

```markdown
## [YYYY-MM-DD] <bug 简述>
- **根因**：xxx
- **触发条件**：xxx
- **修复方式**：xxx
- **预防方法**：xxx
- **回归测试**：`<测试文件路径>#<测试函数名>`
- **转化为规约**：（如需要，说明应更新哪个 harness/ 文件）
- **相关单号**：[TAPD-XXXX]
```

### Step 5：清理与复盘

- 确认原始反馈循环不再复现 bug
- 删除所有临时调试代码（带 `[DEBUG-]` 前缀的日志、临时 harness）
- 如果发现架构问题导致 bug 难以测试，记录到 `ARCHITECTURE.md` 已知技术债
- 思考：什么能预防这个 bug？如果答案涉及架构改进，在 PR 描述中提出建议

## 禁止项

- 不能修改已有测试的断言来让测试通过
- 不能只修复表象（如 catch 异常后忽略），必须找到根因
- 不能在没有回归测试的情况下提交修复代码
- 不能在修复范围外引入无关改动（保持 diff 干净）
- 不能跳过 Step 2 的失败验证（不确认红灯就不能进入 Step 3）

## 输出格式

修复完成后，回复必须包含：

1. **根因说明**：一句话
2. **修改文件列表**：代码文件 + 测试文件
3. **测试结果**：新增测试通过，已有测试全部通过
4. **failures.md 更新内容**：直接粘贴追加的记录
