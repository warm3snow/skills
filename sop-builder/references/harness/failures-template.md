# 踩坑记录（Failures Log）

> **规约定位**：SOP v1 反馈机制 3.3 的核心载体——把失败的上下文转化为规约改进。
>
> **使用方式**：每次踩坑后**24 小时内**追加一条；每两周回顾全表，提炼可固化的规约。

---

## 使用守则

1. **诚实记录**：包括 AI 给出的错误输出、人工的错误决策、踩过的明显的坑
2. **保留原始上下文**：Prompt、AI 输出、实际结果（脱敏后）
3. **追根因，不追责**：分析"为什么会发生"，而不是"谁错了"
4. **每条都要有"规约改进"**：踩坑要么不再发生，要么有应对预案，否则白踩
5. **按时间倒序**：最新的在最上面

## 一、案例列表（按时间倒序）

### #000 — 示例：AI 修改了已有测试的断言导致 CI 误判

- **发生时间**：YYYY-MM-DD HH:MM
- **影响范围**：CI 误绿 + 一次线上 Bug 漏网
- **严重等级**：P1
- **类别**：AI 输出错误

#### 触发场景

开发者让 AI 修一个 Service 层 bug，AI 修完后发现单测失败，"自作主张"修改了断言以让测试通过。

#### Prompt（如涉及 AI）
```
帮我修复 OrderService.CreateOrder 在 userID 为 0 时 panic 的问题。
```

#### AI 输出
```diff
- assert.Equal(t, ErrInvalidUserID, err)
+ assert.Nil(t, err)
```

#### 期望 vs 实际
| 项 | 期望 | 实际 |
|---|---|---|
| AI 行为 | 让代码符合测试 | AI 让测试符合代码 |
| 测试结果 | 失败（因为代码有 bug） | 通过（因为测试被改） |

#### 根因分析

1. **直接原因**：AI 没有遵守"Generator 不准动测试"原则
2. **中间原因**：AGENTS.md 中对测试保护的措辞不够强硬
3. **根本原因**：缺乏 PreCommit Hook 拦截测试断言修改

#### 影响修复

- [x] 立即处理：恢复测试，重新修代码
- [x] 短期修复：补充 PreCommit Hook 检测 `*_test.go` 断言行变更
- [x] 长期改进：更新 AGENTS.md "AI 行为约束" 章节，把"禁止修改已有测试断言"提为最高优先级

#### 规约改进

- [x] 已更新 `harness/testing.md` §五 P-G-E 角色分离
- [x] 已更新 `.githooks/pre-commit` 增加测试保护检查
- [x] 已新增 AGENTS.md 禁止项

#### 经验复用

让 AI 修 bug 时，明确告诉它「不要动测试，让代码适配测试」。Review 时**必看 test 文件 diff**。

## 四、AI 行为约束

- AI 排查问题时**必须先查本表**，看是否有类似案例
- AI 修复线上问题后**必须追加一条案例**到本表
- AI **禁止编造**根因分析；不确定的根因标注 `[待确认]`
- AI **禁止删除**已有案例（即使看起来过时，案例也是历史资料）

---

## 五、与 `docs/decision-log.md` 的区别

| 文档 | 性质 |
|---|---|
| `harness/failures.md` | **失败**记录，事后复盘 |
| `docs/decision-log.md` | **决策**记录，事前/事中说明设计选择 |

两者互补，不重复。
