# 部署与运行规范

> **规约定位**：本服务从构建到运行到部署的硬性规则。
>
> **门禁联动**：部署流水线 Checklist 阻断；灰度未通过禁止全量。

---

## 一、构建规范

### 1.1 构建产物

| 项 | 规则 |
|---|---|
| 二进制 / 镜像 | 必须可复现构建（同 commit → 同产物） |
| 版本号 | 注入 Git Commit Hash 与构建时间 |
| 构建环境 | Docker 镜像或 CI 标准环境，禁止本机直接构建发布 |

### 1.2 镜像规范

| 规则 | 说明 |
|---|---|
| 基础镜像 | 用 `[公司基础镜像 / distroless / alpine]` |
| 多阶段构建 | 编译 + 运行分离，减小镜像体积 |
| 镜像 Tag | `<service>-<version>-<commit-short>` |
| 禁止内容 | `.env` / 密钥 / 调试工具 / shell 调试栈 |

---

## 二、环境分层

| 环境 | 用途 | 配置来源 | 发布门禁 |
|---|---|---|---|
| **dev** | 开发自测 | `config.local.yaml` | 无 |
| **test** | 测试团队 E2E | 配置中心 test 环境 | CI 通过 |
| **staging** | 预发，与生产对齐 | 配置中心 staging 环境 | 测试通过 |
| **prod** | 生产 | 配置中心 prod 环境 | 灰度通过 + 人工验收 |

---

## 三、配置管理

### 3.1 配置分级

| 配置类型 | 存储位置 | 谁可访问 |
|---|---|---|
| 公共配置（端口、超时） | 配置文件 / 配置中心 | 所有人 |
| 业务配置（开关、阈值） | 配置中心 | 开发 + 运维 |
| 敏感配置（密钥、Token、数据库密码） | 配置中心加密 / 密钥管理服务 | 运维 + 安全 |

### 3.2 敏感配置规则（强约束）

- ❌ **禁止**在代码、yaml、commit message 中出现密钥
- ❌ **禁止**在日志中打印密钥
- ❌ **禁止**通过环境变量在 K8s yaml 明文存储（用 Secret）
- ✅ 通过配置中心加密字段或 KMS 注入

### 3.3 配置变更流程

```
修改 config.example.yaml（占位符）→ 提 PR → Review
    ↓
修改配置中心对应环境的配置 → 通知相关人 → 灰度验证
    ↓
全量推送
```

---

## 四、发布流程

### 4.1 标准发布

```
1. PR 合并 main
2. CI 触发构建
3. 镜像推送制品库
4. 发布到 test 环境 → 测试团队验证
5. 发布到 staging 环境 → 预发验证
6. 灰度发布到 prod（按比例：1% → 10% → 50% → 100%）
7. 每轮灰度观察 [N] 分钟
8. 全量后保持监控 [M] 分钟
```

### 4.2 发布 Checklist（强约束）

发布前必须勾选：

- [ ] CI 全部通过
- [ ] 测试团队验收通过
- [ ] 涉及 schema 变更 → 已先发布 migration
- [ ] 涉及兼容性 → 下游服务已确认
- [ ] 涉及配置变更 → 配置中心已更新
- [ ] 回滚方案已准备
- [ ] 监控告警已就绪
- [ ] 灰度策略已制定
- [ ] 关联 TAPD / Jira 单号

## 五、灰度策略

### 5.1 灰度维度

| 维度 | 适用场景 |
|---|---|
| 按实例比例 | 通用场景 |
| 按用户 ID 取模 | 用户态服务 |
| 按地域 | 多地域部署 |
| 按租户 | SaaS 多租户 |

### 5.2 灰度观察指标

- 业务指标：成功率 / QPS / 关键业务量
- 系统指标：CPU / 内存 / Goroutine / GC
- 错误指标：错误率 / 错误日志数 / 告警数
- 性能指标：P50 / P99 延迟

### 5.3 灰度异常判定

| 指标 | 异常阈值 | 处理 |
|---|---|---|
| 错误率上升 | > [X]% | 立即回滚 |
| P99 延迟上升 | > [X]% | 暂停灰度，分析 |
| 业务量下降 | > [X]% | 暂停灰度，分析 |

## 六、回滚

### 6.1 回滚条件

灰度或全量后出现：
- 业务指标显著恶化
- 错误率超过阈值
- 上游或下游主动反馈

### 6.2 回滚方式

| 类型 | 方式 |
|---|---|
| 代码回滚 | 切换到上一个稳定镜像 |
| 配置回滚 | 配置中心切回上一版本 |
| Schema 回滚 | **数据库 schema 通常不回滚**，通过代码兼容性处理 |
| 数据回滚 | 走 DBA 评审，谨慎处理 |

### 6.3 回滚后

- 必须**复盘**：根因 + 改进项
- 在 [`harness/failures.md`](./failures.md) 记录
- 必要时更新规约

## 七、健康检查

### 7.1 必备接口

| 接口 | 用途 | 检查内容 |
|---|---|---|
| `GET /health` | K8s livenessProbe / 监控 | 进程存活 |
| `GET /ready` | K8s readinessProbe | DB / 依赖 / 配置 都就绪 |
| `GET /metrics` | Prometheus | 业务/系统指标 |

### 7.2 健康检查响应

```json
{
  "status": "ok",
  "version": "v1.2.3",
  "commit": "abc1234",
  "uptime_seconds": 3600,
  "dependencies": {
    "mysql": "ok",
    "redis": "ok",
    "kafka": "ok"
  }
}
```

## 八、监控与告警

### 8.1 必备监控

| 类别 | 指标 |
|---|---|
| 业务 | 核心接口 QPS / 成功率 / 业务量 |
| 接口 | P50 / P99 延迟、错误率 |
| 系统 | CPU / 内存 / Goroutine / GC |
| 依赖 | DB 连接池、Redis 命中率、MQ 消费延迟 |
| 错误 | ERROR / FATAL 日志数 |

### 8.2 告警分级

| 级别 | 触发条件 | 通知方式 | 响应时效 |
|---|---|---|---|
| P0 | 服务不可用 | 电话 + IM | 5 分钟 |
| P1 | 核心功能受损 | IM | 30 分钟 |
| P2 | 性能劣化 | IM | 2 小时 |
| P3 | 监控异常 | 邮件 | 1 工作日 |

详细告警处理见 [`docs/runbook.md`](../docs/runbook.md)。

---

## 九、运行时禁止行为

- ❌ **禁止**在线上直接 `exec` 改文件
- ❌ **禁止**绕过流水线直接发版
- ❌ **禁止**在 prod 跳过灰度
- ❌ **禁止**在线上手工执行未经评审的 SQL
- ❌ **禁止**在生产关闭告警

---

## 十、AI 行为约束

- AI 修改部署相关配置必须**走 PR**，不能直接改线上配置
- AI 不得**编造**部署步骤
- AI 涉及生产环境操作必须**停下来与用户确认**
- AI 修改 schema 后必须提示**先发布 migration，再发布代码**
