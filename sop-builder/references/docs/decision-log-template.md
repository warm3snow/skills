# 设计决策日志（ADR Log）

> **文档定位**：完整记录历史设计决策（ADR = Architecture Decision Record），防止 AI 或新员工"好心重构"掉关键设计。
>
> **与 [`harness/failures.md`](../harness/failures.md) 的区别**：
> - ADR 是**事前/事中**的设计选择记录
> - failures 是**事后**的失败复盘记录

---

## 使用规则

1. **何时写 ADR**：架构级决策、技术选型、设计模式选择、跨服务契约、性能/容量决策
2. **何时不写**：实现细节、命名风格、代码格式（这些归 [`harness/coding-style.md`](../harness/coding-style.md)）
3. **ADR 不可删除**：即使决策被推翻，原 ADR 状态改为「已废弃」，新 ADR 引用旧 ADR
4. **编号连续**：ADR-001、ADR-002，...，**不跳号**
5. **状态明确**：已提议 / 已采纳 / 已废弃 / 已被替代

## ADR-001: [示例：订单状态机集中管理]

| 项 | 内容 |
|---|---|
| 日期 | YYYY-MM-DD |
| 状态 | 已采纳 |
| 作者 | `[待确认]` |
| 评审人 | `[待确认]` |
| 相关 ADR | - |

### 背景

`[早期订单状态判断散落在多处（handler、service、定时任务），导致状态机演进时多处遗漏。]`

### 决策驱动因素

- 状态机演进频繁
- 业务规则需要可审计
- 防止状态非法跳转

### 考虑的方案

#### 方案 A：散落各处（现状）

- **优点**：开发简单，临时改快
- **缺点**：长期维护成本高，易遗漏
- **成本**：0

#### 方案 B：集中状态机 + 状态流转表

- **优点**：演进集中、可审计、防止非法跳转
- **缺点**：初次改造成本高
- **成本**：5 人天

#### 方案 C：引入第三方状态机库

- **优点**：开箱即用
- **缺点**：引入新依赖，团队学习成本
- **成本**：3 人天

### 最终决策

**选择：方案 B**

**核心理由**：

1. 状态机是订单核心，集中管理便于演进
2. 第三方库的灵活度低于自研，长期看自研可控
3. 改造成本可控

### 影响与约束

**正面影响**：
- 新增状态只需改 `order_state.go` 一处

**负面影响**：
- 已有代码需大幅重构

**新增约束**：
- ❌ 禁止在 service 层以外的地方判断/修改订单状态
- ✅ 所有状态流转必须通过 `OrderStateMachine.Transit()` 方法

### 实施计划

- [x] 阶段 1：设计状态机结构（YYYY-MM-DD）
- [x] 阶段 2：迁移现有代码（YYYY-MM-DD）
- [x] 阶段 3：补充测试覆盖（YYYY-MM-DD）

### 验证与回滚

- **验证指标**：状态机错误日志数 < 1/天
- **回滚条件**：错误数 > 100/天且无法定位
- **回滚方案**：保留旧分支，切回原实现

## 废弃 ADR 处理

当某个 ADR 被新 ADR 替代时：

1. 保留原 ADR 内容不变
2. 状态改为「已被 ADR-YYY 替代」
3. 在「相关 ADR」字段引用新 ADR
4. 新 ADR 的「背景」章节必须说明：为什么之前的决策不再适用

---

## AI 行为约束

- AI **不得删除**已有 ADR
- AI **修改架构关键设计前**，必须先读对应 ADR
- AI 提出新设计方案时，应**预先生成 ADR 草稿**供人工评审
- AI 不得**编造** ADR 内容
