## 场景 1：添加新的 HTTP/RPC 接口

🎯 目标：对外暴露一个新的接口能力

### 📍 定位

| 步骤 | 去哪里 | 做什么 |
|------|-------|-------|
| 1 | `internal/handler/` | 添加 handler 函数，解析请求参数 |
| 2 | `internal/service/` | 实现业务逻辑 |
| 3 | `internal/repo/`（如需读写数据） | 添加数据访问方法 |
| 4 | `internal/model/`（如需新数据结构） | 定义请求/响应 struct |
| 5 | 路由注册文件 | 注册新路由 |
| 6 | `docs/api.md` | 补充接口文档 |

### 📝 操作步骤

```
1. [在 handler 层定义接口签名，参考已有 handler 的写法]
2. [在 service 层实现业务逻辑，注意依赖注入方式]
3. [如需数据库操作，在 repo 层添加方法，遵循现有命名规范]
4. [在路由文件中注册路由，注意路由分组和中间件]
5. [编写测试用例]
6. [更新 docs/api.md]
```

### ⚠️ 常见陷阱

- 不要在 handler 层写业务逻辑，handler 只负责参数解析和响应组装
- 新增接口必须补充 `docs/api.md`，否则其他开发者（和 AI）不知道接口存在
- 如果接口涉及跨服务调用，必须先确认契约兼容性，参考 `docs/api.md` 中的兼容性要求

### ✅ 完成检查

- [ ] handler 层只做参数解析，业务逻辑在 service 层
- [ ] 测试用例已添加，`make test` 通过
- [ ] `docs/api.md` 已更新
- [ ] 路由已注册，可通过 `make run` 本地验证

## 场景 2：修改已有接口的行为

🎯 目标：调整某个已有接口的返回值、业务逻辑或参数处理

### 📍 定位

| 步骤 | 去哪里 | 做什么 |
|------|-------|-------|
| 1 | `docs/api.md` | 先了解当前接口定义和调用方 |
| 2 | `internal/handler/` | 找到 handler，定位到 service 调用 |
| 3 | `internal/service/` | 找到业务逻辑实现 |
| 4 | `internal/model/` | 确认请求/响应 struct |

### 📝 操作步骤

```
1. [先阅读 docs/api.md 中该接口的定义，了解调用方和兼容性要求]
2. [定位到 service 层的具体实现，阅读现有逻辑和注释]
3. [评估修改是否影响响应结构：如果增加字段 → 兼容；如果删除/改字段 → 必须走 RFC]
4. [实施修改]
5. [更新 docs/api.md 中的接口文档]
6. [检查并补充测试用例]
```

### ⚠️ 常见陷阱

- **兼容性陷阱**：删除字段或修改字段语义是不兼容变更，必须通知所有消费方
- **隐性依赖**：修改 service 层逻辑可能影响其他调用同一 service 方法的 handler
- **枚举值变更**：新增枚举值必须确认所有消费方的兜底逻辑

### ✅ 完成检查

- [ ] 确认修改是兼容的，或者已与消费方确认
- [ ] `make test` 通过，旧测试未被破坏
- [ ] `docs/api.md` 已更新
- [ ] 如果是不兼容变更，已通知下游服务负责人

## 场景 3：给已有数据模型添加新字段

🎯 目标：在已有 struct / 数据库表中新增一个业务字段

### 📍 定位

| 步骤 | 去哪里 | 做什么 |
|------|-------|-------|
| 1 | `internal/model/` | 找到对应 struct，添加新字段 |
| 2 | 数据库 migration 目录 | 添加 migration 文件（只追加，不修改已有） |
| 3 | `internal/repo/` | 确认 SQL 查询是否需要更新 |
| 4 | `internal/service/` | 确认业务逻辑是否需要使用新字段 |
| 5 | `internal/handler/` | 确认 API 响应是否需要返回新字段 |
| 6 | `docs/domain-model.md` | 更新字段说明 |

### 📝 操作步骤

```
1. [在 model struct 中添加新字段，注意 json/db tag 命名规范]
2. [编写 migration 文件，只允许 ADD COLUMN，禁止修改已有列]
3. [检查 repo 层 SQL，确认新字段是否需要加入 SELECT/INSERT/UPDATE]
4. [在 service 层补充新字段赋值逻辑]
5. [在 handler 层确认是否需要返回新字段给调用方]
6. [更新 docs/domain-model.md，说明新字段的业务含义]
7. [编写测试用例]
```

### ⚠️ 常见陷阱

- 新字段必须给默认值或标记为可选，否则旧数据会报错
- migration 只能追加，**绝对不能修改已有的 migration 文件**
- 数据库字段的业务含义必须记录到 `docs/domain-model.md`，否则后续开发者无法理解
- 如果新字段需要被其他服务消费，必须走契约变更流程

### ✅ 完成检查

- [ ] 新字段有合理默认值或为可选
- [ ] migration 文件已添加，未修改已有 migration
- [ ] `make test` 通过
- [ ] `docs/domain-model.md` 已更新，字段业务含义已说明
- [ ] 如果字段涉及跨服务契约，已通知消费方

## 场景 4：添加新的 MQ 消费者或生产者

🎯 目标：新增消息的发送或消费逻辑

### 📍 定位

| 步骤 | 去哪里 | 做什么 |
|------|-------|-------|
| 1 | `internal/service/` | 实现消息生产/消费的业务逻辑 |
| 2 | `config/` | 确认 MQ 配置（Topic、Group、Broker） |
| 3 | MQ 注册/订阅文件 | 注册新的消费者或生产者 |
| 4 | `docs/api.md` MQ 章节 | 补充消息契约文档 |

### 📝 操作步骤

```
1. [确认 Topic 名称、消息格式和消费组]
2. [在 service 层实现生产/消费逻辑]
3. [在 MQ 注册文件中注册消费者/生产者]
4. [消费端必须实现幂等逻辑，防止重复消费]
5. [更新 docs/api.md 中的 MQ 契约部分]
6. [编写测试用例，覆盖正常和异常场景]
```

### ⚠️ 常见陷阱

- **幂等性**：MQ 消息可能重复投递，消费端必须幂等
- **消费顺序**：如果业务依赖消息顺序，必须确认 MQ 的顺序消费保障
- **向后兼容**：新增消息字段允许，但修改已有字段语义或删除字段必须走 RFC
- **消费延迟**：新增消费者后注意消费延迟监控

### ✅ 完成检查

- [ ] 消费端实现了幂等逻辑
- [ ] `docs/api.md` 中 MQ 契约已更新
- [ ] `make test` 通过
- [ ] 本地可正常生产和消费消息

## 场景 5：修改配置项

🎯 目标：新增或调整服务配置

### 📍 定位

| 步骤 | 去哪里 | 做什么 |
|------|-------|-------|
| 1 | `config/config.yaml` | 修改配置值 |
| 2 | `config/config.example.yaml` | 同步更新示例配置 |
| 3 | 配置 struct 定义文件 | 如新增配置项，添加对应 struct 字段 |

### 📝 操作步骤

```
1. [在 config struct 中添加新字段（如需要）]
2. [在 config.yaml 中添加配置项]
3. [在 config.example.yaml 中同步添加，并写注释说明]
4. [在 service 中使用新配置项，替换硬编码值]
5. [确认本地 make run 正常启动]
```

### ⚠️ 常见陷阱

- 不要忘记同步更新 `config.example.yaml`，否则其他开发者本地起不来
- 配置变更如果影响线上行为，需要确认灰度策略
- 敏感配置（密钥、token）不要写在 yaml 中，走环境变量或配置中心

### ✅ 完成检查

- [ ] `config.example.yaml` 已同步更新
- [ ] 本地 `make run` 正常启动
- [ ] 敏感配置未硬编码在文件中

### 📝 操作步骤

```
1. [确认错误现象：错误码、日志关键词、影响范围]
```
