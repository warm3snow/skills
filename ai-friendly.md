## 微服务仓库 AI 友好化指南

### 核心思路

AI（无论是 Copilot、Cursor 还是 Claude Code）理解代码仓库的方式和人类不同——它需要**结构化、上下文丰富、边界清晰**的信息。目标是让 AI 在「最少提问」的情况下能独立完成任务。

---

## 一、必备文档体系（优先级排序）

### 🔴 P0 — 必须有

#### 1. `README.md` — 仓库门面
```markdown
# 服务名称
## 这个服务是干什么的（1句话）
## 业务职责边界（它负责什么、不负责什么）
## 技术栈（语言/框架/中间件版本）
## 快速启动（本地跑起来的最短路径）
## 依赖的上下游服务
## 关键配置项说明
```

#### 2. `AGENTS.md` / `CLAUDE.md` — AI 专属指导文件
这是**最关键的 AI 友好化动作**，专门写给 AI Agent 看的：
```markdown
# AI Agent 指南

## 仓库架构一句话描述
## 核心模块划分（目录树 + 每个目录的职责）
## 扩展新功能的标准路径（step by step）
## 禁止触碰的文件/模块（及原因）
## 代码风格约定
## 测试规范（如何跑测试、测试覆盖要求）
## 常见任务示例（添加接口/修改数据模型/接入新中间件）
```

> 💡 Claude Code、Cursor、Codex 等 agent 会自动读取 `AGENTS.md` / `CLAUDE.md`，优先写这个文件。

---

### 🟡 P1 — 强烈建议

#### 3. `docs/architecture.md` — 架构文档
- 服务在整体系统中的位置（可以是 ASCII 图或 Mermaid 图）
- 核心数据流向
- 关键设计决策及原因（ADR 格式）

#### 4. `docs/domain-model.md` — 领域模型
- 核心业务实体定义（用中文！让新员工和 AI 都能理解业务语义）
- 实体关系
- 关键业务规则

#### 5. `docs/api.md` 或 OpenAPI Spec
- 对外暴露的接口（入参/出参/错误码/业务含义）
- gRPC/消息队列 schema 同样需要文档化

---

### 🟢 P2 — 进阶

#### 6. `docs/runbook.md` — 操作手册
- 常见故障排查路径
- 告警含义解释
- 数据修复操作

#### 7. `docs/decision-log.md` — 决策日志
记录"为什么这么做"，防止 AI 或新员工"好心"重构掉关键设计

## 二、代码本身的 AI 友好化

| 维度 | 做法 |
|------|------|
| **命名** | 业务含义命名，避免缩写（`calcTaxAmt` → `calculateTaxAmount`） |
| **注释** | 复杂逻辑写"为什么"而不是"做什么" |
| **接口注释** | 每个 public 方法/接口写清楚：输入含义、输出含义、异常情况 |
| **模块边界** | 单一职责，减少隐式依赖 |
| **TODO/FIXME** | 标注清楚背景，AI 才能判断是否可以处理 |

---

## 三、落地步骤建议

```
第1步（1-2天）：先做一个服务作为模板
  └─ 写 AGENTS.md + 补全 README

第2步（1周）：用 AI 批量生成初稿
  └─ 让 Claude/Copilot 读代码，生成文档草稿
  └─ 人工校对业务语义部分

第3步（持续）：把文档更新纳入 MR checklist
  └─ 改了接口/数据模型 → 必须更新对应文档
```

---

## 四、一个实用技巧

在 `AGENTS.md` 里加一段**"如何扩展新功能"的 step-by-step**，例如：

```markdown
## 添加新的业务接口（标准路径）
1. 在 `proto/` 目录定义接口 schema
2. 在 `internal/handler/` 添加 handler
3. 在 `internal/service/` 实现业务逻辑
4. 在 `internal/repo/` 添加数据访问层（如需要）
5. 在 `test/` 添加集成测试
6. 更新 `docs/api.md`
```

这段内容对新员工和 AI Agent 都是**最高价值**的信息——它把隐性的架构约定变成了显性的操作流程。

## 五、跨服务 / 跨团队 AI 协作

当微服务分属不同小组、无法由一个 AI 统一改代码时，AI 友好化目标应从“统一编码”调整为“统一上下文、统一影响面分析、统一任务拆分”。

### 推荐新增一个全局上下文仓库

```text
ai-system-context/
├── SYSTEM.md                    # 全局系统总览
├── SERVICE-CATALOG.md           # 服务目录：服务归属、职责、仓库、数据归属
├── SERVICE-GRAPH.md             # 服务依赖图：HTTP/RPC/MQ/DB/外部系统
├── CONTRACTS.md                 # 跨服务契约索引：接口、RPC、MQ、字段兼容性
├── FEATURE-ROUTING.md           # 功能路由表：某类需求通常涉及哪些服务
├── CHANGE-PLAYBOOKS.md          # 跨服务变更 SOP
├── VIBE-CODING-PROMPTS.md       # 可复制给各小组 AI 的 prompt
├── rfcs/                        # 跨服务功能 RFC
├── tasks/                       # 按小组/服务拆分的任务单
└── services/                    # 每个服务一张服务卡片
```

### 核心原则

| 原则 | 说明 |
|------|------|
| **不越权改代码** | 全局 AI 只生成 RFC、影响面分析、任务单和 prompt，代码仍由各小组负责 |
| **先契约后实现** | 涉及接口、RPC、MQ、字段语义变化时，先定义兼容策略 |
| **服务卡片最小化** | 每个服务只需维护职责边界、对外能力、数据归属、负责人和变更注意事项 |
| **任务单按小组拆分** | 每个小组只收到与自己服务相关的输入、输出、验收标准 |

### 一个跨服务需求的推进方式

```text
第1步：用 SERVICE-CATALOG.md 和 FEATURE-ROUTING.md 判断涉及服务
第2步：用 SERVICE-GRAPH.md 和 CONTRACTS.md 分析调用链和契约风险
第3步：生成 rfcs/<feature>.md，明确目标、非目标、数据流、契约变更和发布顺序
第4步：生成 tasks/<feature>/<service-or-team>.md，拆给各小组
第5步：各小组在自己的仓库里结合 AGENTS.md 做 vibe coding
第6步：联调后回填 CONTRACTS.md、FEATURE-ROUTING.md 和服务卡片
```

### 给 vibe coding 的正确输入

不要直接说“帮我把这个跨三个服务的功能做了”。应先生成任务单，再把任务单交给对应小组的 AI：

```markdown
你现在只负责 `order-service`，不要修改其他服务。
请阅读本仓库 `AGENTS.md`、全局 RFC 和本服务任务单。
先输出：
1. 当前仓库中相关模块和文件位置；
2. 是否需要接口契约变更；
3. 兼容性风险；
4. 修改计划和测试计划。
确认后再实现。
```

一句话：**单仓库 AGENTS.md 解决“这个服务内部怎么改”；全局 ai-system-context 解决“这个需求该找哪些服务、哪些小组、按什么顺序推进”。**
